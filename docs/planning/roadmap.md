# Project Roadmap ‚Äì Phases & Stages

WobbleBot‚Äôs development is organized into **five phases**, each containing **five stages**.  We build like a house: lay the foundation, frame the structure, wire up systems, finish the surfaces, then polish and decorate.  This roadmap lays out what gets built when; it is a *guiding structure*, not a rigid contract‚Äîphases and stages may be merged or adjusted as we learn.

## Phase¬Ý1 ‚Äì Foundation & Sandbox ‚úÖ Complete (2026-05-13)

**Goal:** Bootstrapped skeleton of WobbleBot with no real trading risk.

1. **Stage¬Ý1.1 ‚Äì Repo & Scaffolding** ‚úÖ (2025-11-24) ‚Äì Create the repository structure (`src/`, `docs/`, `config/`, `docker/`).  Add base Python project configuration, linting, and formatting tools.
2. **Stage¬Ý1.2 ‚Äì Hex Core Skeleton** ‚úÖ (2025-11-24) ‚Äì Define core domain models (`Order`, `Trade`, `Position`).  Define abstract ports (e.g., `ExchangePort`, `StoragePort`, `AdvisorPort`, `HarvesterPort`, `NotifierPort`).
3. **Stage¬Ý1.3 ‚Äì Storage & Logging Backbone** ‚úÖ (2026-05-12) ‚Äì Implement the SQLite adapter and configure logging.  Provide basic persistence for trades, configuration snapshots, and events.
4. **Stage¬Ý1.4 ‚Äì Kraken Mock & Simulation Mode** ‚úÖ (2026-05-12) ‚Äì Implement a fake exchange adapter for dry‚Äërun simulations.  Support a paper trading loop with hard‚Äëcoded scenarios.
5. **Stage¬Ý1.5 ‚Äì Phase¬Ý1 Integration Check** ‚úÖ (2026-05-13) ‚Äì Demonstrate an end‚Äëto‚Äëend simulated cycle: load config ‚Üí run core loop against the mock exchange ‚Üí persist results ‚Üí view logs.  No external API calls yet.

## Phase 2 ‚Äì Core Trading Engine (Real Kraken, No Money Moves)

**Goal:** Deterministic micro-grid trading against **real Kraken** with tiny exposure and **no withdrawals**.

1. **Stage¬Ý2.1 ‚Äì Kraken Adapter (Read‚ÄëOnly + Minimal Data Collector)** ‚úÖ (2026-05-14) ‚Äì Integrate with the Kraken API using a read‚Äëonly key.  Fetch tickers, order books, and account balances.  Provide a minimal `DataCollector¬Ýv1` that supplies current prices and balances to the Bot¬ÝCore.  *Read paths only (Ticker + BalanceEx). Order placement, OpenOrders, and TradesHistory parsing deferred to Stage¬Ý2.3. Live integration verified via `python -m wobblebot.cli.status`.*
2. **Stage 2.2 ‚Äì Micro-Grid Engine** ‚úÖ (2026-05-14) ‚Äì Implement the configurable grid logic per asset (grid boundaries, spacing, order sizing).  Enforce per‚Äëcoin caps on maximum orders and maximum funds in play.  *Five slices landed: config schemas (`GridConfig`, `SafetyConfig`, YAML loader), pure grid math (`compute_grid_levels`, `next_counter_action`, `is_offside`), `GridEngine` service with `GridState` persistence, safety cap enforcement (per-coin/total exposure + daily-spend), end-to-end integration test (1000-tick oscillation, 500 cycles, positive realized P&L). Six ratified design decisions in ADR-006. Counter orders match filled-order base amounts. Engine wires to `MockExchangeAdapter`; Stage 2.3 swaps in real Kraken via the `ExchangePort` contract.*
3. **Stage¬Ý2.3 ‚Äì Live Paper Mode / Tiny‚ÄëSize Mode** ‚úÖ (2026-05-14) ‚Äì Enable live trading against Kraken with minimal order sizes (or full paper mode via configuration).  Track profit and loss, cycle counts, and basic volatility metrics.  *Five slices landed: KrakenAdapter trading methods + AssetPairs precision cache, live integration tests via validate=true, cli/preflight diagnostic CLI, cli/live operational CLI with hard caps + clean SIGINT shutdown, plus stage-close operator handoff docs. Verified live with tools/first_real_trade.py: real-money round-trip on the operators account cost $0.08 (2x 0.40% taker fee), 148ms fill latency, perfect cleanup. Live taker fee 0.40% vs mocks 0.26% ‚Äî only matters for marketable orders; engines normal maker-side use case unaffected.*
4. **Stage¬Ý2.4 ‚Äì Multi‚ÄëAsset Support** ‚úÖ (2026-05-14) ‚Äì Extend the core to run grids for multiple whitelisted coins (e.g., DOGE, ADA, SOL, MATIC, ETH).  Enforce shared safety rules such as daily spend caps and global exposure limits.  *cli/live takes --symbols comma-separated; each tick steps every symbol in series through the same GridEngine. Per-symbol step errors swallowed at the CLI so one bad coin cannot kill the session. Caps: total/daily-spend global across symbols; per-coin scoped per symbol. Engine layer required ZERO changes (every per-coin entity already keys by symbol, hex layer purity paid off again). Five new multi-coin engine tests; 296 unit tests pass total.*
5. **Stage 2.5 ‚Äì Phase 2 Integration Check** ‚úÖ (2026-05-14) ‚Äì Demonstrate a full pipeline: configuration ‚Üí live Kraken adapter + `DataCollector v1` ‚Üí micro-grid engine ‚Üí logs and database entries.  All withdrawals remain disabled at the API level.  *Live multi-coin verification: cli/live --symbols BTC/USD,ETH/USD ran 304.6s against the operators account, 54 ticks per coin, 0 fills, 6/6 open orders cleanly cancelled on runtime-cap shutdown, session PnL $0.0000. Closing summary at docs/planning/phase-2-summary.md. Phase 2 total real-money cost across both verifications: $0.08.*

## Phase 5 ‚Äì Operator Interaction Engine

**Goal:** A bidirectional Discord interface with multi-turn conversational LLM intent parsing, structured command + query catalog, and ADR-002-preserving confirm-before-execute. `cli/live` stays Discord-ignorant; a new `cli/operator` daemon owns the chat surface and communicates with the engine through SQLite tables in a new `operator.db`. Reframed from the original Phase 5 (dashboard + notifier + control + reliability + maintenance + performance + v1.0 release) on 2026-05-16 after the operator surfaced a broader vision; per ADR-013. The displaced original-Phase-5 stages reorganize into Phases 6‚Äì8 below.

**Architectural reference:** [ADR-013 ‚Äî Operator Interaction Engine](../architecture/decisions.md#adr-013--operator-interaction-engine-discord--conversational-llm).

1. **Stage 5.1 ‚Äì Operator Domain & Ports** ‚Äì Pure-domain stage. `OperatorPort` + `AssistantPort` ABCs; `OperatorIntent` typed sum (`Command | Query | Conversational | Unparseable`) using Pydantic discriminated unions; concrete `OperatorCommand` catalog (`Pause`, `Resume`, `PauseAll`, `ResumeAll`, `CancelOpenOrders`, `Stop`) and `OperatorQuery` catalog (`Status`, `OpenOrders`, `RecentFills`, `RecentSuggestions`, `RecentNews`, `HarvesterStatus`, `RecentProposals`, `GridConfig`, `Help`) with per-query `*Result` types; `PendingCommand` audit-trail model; `ConversationTurn` + `ConversationContext` for multi-turn prompt assembly; `OperatorError` + `AssistantError` exceptions. No I/O. No Discord. No LLM call. No SQLite table. Three slices: 5.1.A operator types + port, 5.1.B assistant types + port, 5.1.C close. Per [stage-5.1-design.md](stage-5.1-design.md).
2. **Stage 5.2 ‚Äì Discord Transport Adapter** ‚Äì `discord.py`-backed bot client in `adapters/discord_transport.py`. Gateway lifecycle (connect, reconnect, heartbeat). Inbound message handler (allowlist-filtered) emits a `ConversationContext`. Reaction handler for confirm / reject on confirmation embeds. Bot-identity helpers (typing indicator, embed builders for command summaries, status reports, error messages). Works against a stubbed `OperatorPort` and stubbed `AssistantPort` so this stage can ship without the LLM or the engine integration landing.
3. **Stage 5.3 ‚Äì Operator Assistant (Ollama)** ‚Äì `OllamaAssistantAdapter` implementing `AssistantPort`. New `config/prompts/operator.md` system prompt (markdown body + YAML frontmatter, same pattern as the four MoE prompts). Multi-turn context-window assembly (default 10 turns, configurable). JSON-output schema enforcement ‚Äî the LLM must emit an `OperatorIntent` payload that round-trips through Pydantic validation. Token / cost budgeting on a per-turn basis. Verification: live-tested against the operator's local Ollama install with a handful of sample prompts.
4. **Stage 5.4 ‚Äì Engine Integration** ‚Äì Engine gains `pause_symbol(symbol)`, `resume_symbol(symbol)`, `cancel_open_orders(symbol)`, and `request_stop()` methods. New `services/operator_service.py` implements `OperatorPort` ‚Äî dispatches approved `PendingCommand`s to engine ops and composes `QueryResult`s from `StoragePort` reads. **First SQLite table lands here:** `pending_commands` (with full status enum + audit columns). `cli/live` gains a per-tick poll of `pending_commands WHERE status = 'approved'`, dispatches them, marks `dispatched` or `failed`. ADR-002 confirm-before-execute enforced here ‚Äî there is no code path from `OperatorService.dispatch_command` to the engine that does not require `status == 'approved'`.
5. **Stage 5.5 ‚Äì Outbound Notifications** ‚Äì `SqliteNotifierAdapter` implementing `NotifierPort`; writes structured `Notification` rows to a new `notifications` table. `cli/live` injection wired for fill events, cap-trip events, session-start / session-end summaries with PnL. `cli/harvest` injection wired for proposal-generated, withdrawal-executed, withdrawal-failed events. `cli/operator`'s notification forwarder (built in 5.6) reads new rows and posts to Discord. `cli/live` and `cli/harvest` remain Discord-ignorant ‚Äî no `discord.py` import, no bot-token awareness.
6. **Stage 5.6 ‚Äì `cli/operator` Daemon** ‚Äì The long-running CLI that ties Discord + Assistant + OperatorPort together. Maintains the Gateway connection; conversation loop (message in -> parse intent -> confirm-if-command / execute-if-query / reply-if-conversational); notification forwarder (poll `notifications`, post to channel); confirmation-flow handler (manage `pending_commands` status transitions on confirm / reject). New `OperatorConfig` schema in `config/cli.py` covers `bot_token_env`, channel + user allowlists, context window, confirm TTL, schedule cadences. **`conversation_turns` table** lands here. Per-cycle fault isolation: one bad message or one LLM failure never kills the loop. `tools/show_pending.py` operator inspection script for outstanding confirmations.
