# WobbleBot — Incident Runbook

*Operator-facing. The "something broke, now what" checklist — sibling to
[`v1.0-soak-runbook.md`](v1.0-soak-runbook.md) (which covers running a clean soak).*

**This is a *runbook*, not an "incident-response process."** A 2026-05-23 security
audit flagged "no documented IR process" as an L3 gap; "process" implies on-call
rotations, paging, war rooms, blameless post-mortems — none of which apply to a
solo, personal trading tool. This is the solo-shaped version: one doc, no
organization implied, covering the realistic incident scenarios for a
single-operator deployment.

**Three principles for every scenario below:**

1. **Stop the bleeding before you assess.** Halt the affected daemon first; figure
   out what happened second. A stopped bot can't make it worse.
2. **Rotate first, decide cleanup second.** If a secret might be exposed, rotate it
   *now*. Rotation makes the leaked value worthless, which buys you time to decide
   whether history rewriting is worth it.
3. **Document so future-you knows.** A one-line note (what, when, vector, fix) in
   the project's status file or a commit message is enough. You are the whole team.

Each scenario uses the same five sections: **Detection** (how you'd notice) ·
**Stop-the-bleeding** (immediate halt) · **Assess** (what was exposed) · **Recover**
(rotate + fix + restart) · **Document**.

> Safety mechanisms this runbook leans on: the **ADR-003 key separation** (the trade
> key cannot withdraw — only the Harvester key can), now machine-verifiable via
> `cli/preflight` (exit 3 on violation); the **ADR-021 dead man's switch** (Kraken
> auto-cancels open orders if `cli/live` goes silent); the **`.githooks/pre-commit`**
> guard (gitleaks + PII + author-identity); and the **soak-runbook's Abort + restart
> procedure** for engine recovery.

## 1. Suspected API key / secret compromise

A credential may be leaked: a Kraken key (reader / trade / harvester), the Discord
bot token, or `WOBBLEBOT_WEB_SESSION_SECRET`.

- **Detection:** orders or withdrawals on Kraken you didn't initiate; unfamiliar
  Discord bot activity; unexpected web logins (`users.last_login_at`); or the secret
  itself appearing somewhere public (a screenshot, a paste, a commit).
- **Stop-the-bleeding:** stop the affected daemon(s) immediately — SIGINT in the
  foreground, or `pm2 stop` / `systemctl stop` / Portainer stop. If you're unsure
  which key, stop everything (`cli/live`, `cli/harvest`, `cli/operator`, `cli/web`).
- **Assess — what can the leaked credential actually do?**
  - **Trade key** → place/cancel orders only. It **cannot withdraw** (ADR-003;
    `cli/preflight` proves it). Worst case: nuisance orders, bounded by the safety caps.
  - **Harvester key** → withdraw funds. This is the dangerous one — see scenario 2.
  - **Reader key** → read-only (balances, prices). Low blast radius.
  - **Discord token** → impersonate the bot in your server. No money path.
  - **`WOBBLEBOT_WEB_SESSION_SECRET`** → forge dashboard sessions. The dashboard is
    read-mostly; mutations still pass the `pending_commands` confirm firewall (ADR-002).
- **Recover:** rotate the credential at its source —
  - Kraken: Pro → Settings → API → delete the key, mint a fresh one with the **same
    scopes** (for a trade key, **Withdraw OFF**; for harvester, **Withdraw ON, Trade OFF**).
  - Discord: Developer Portal → reset the bot token.
  - Web secret: regenerate (`python -c "import secrets; print(secrets.token_urlsafe(32))"`).
    Rotating it **invalidates every existing session** by design.
  - Update `.env`, restart the daemon. **Re-run `cli/preflight`** to confirm a new
    Kraken trade key is correctly scoped (no withdraw).
- **Document:** leak vector + rotation date.

## 2. Unexpected withdrawal observed in Kraken

A withdrawal appears in Kraken's Funding history that you didn't authorize.

- **Detection:** Kraken Pro → Funding → Withdrawals shows a transfer you didn't
  initiate; or a `transfer_results` row you don't recognize; or a real withdrawal with
  **no** matching `transfer_results` row.
- **Stop-the-bleeding:** stop `cli/harvest` immediately. It is the **only** money-out
  path (ADR-003/004); halting it closes the bot-side door.
- **Assess:** audit the withdrawal — `python tools/show_transfers.py` cross-referenced
  against Kraken's Funding history. Every legitimate withdrawal is operator-initiated
  via `cli/harvest --execute` (seven defense layers) and leaves a `transfer_results`
  row. A withdrawal with no row, or one you don't remember approving, is the alarm.
  Note: the **trade key cannot withdraw**, so a rogue withdrawal points at either the
  **harvester key** being compromised or a **Kraken-side** issue — not the bot's
  trading path.
- **Recover:** if it was **not** operator-initiated, treat the harvester key as
  compromised — rotate it (scenario 1) and **contact Kraken support** for an
  unauthorized withdrawal. Consider whitelist/IP restrictions on the new key.
- **Document:** the refid/txid + your determination (operator / harvester-key / Kraken).

## 3. `cli/web` exposed to the internet unexpectedly

The dashboard, meant to be LAN/loopback-only (ADR-016), became internet-reachable.

- **Detection:** `cli/web`'s startup **WARNING** about a non-loopback bind;
  `users.last_login_at` shows logins from unfamiliar times/places; or you find an open
  port-forward / a reverse-proxy rule you didn't intend.
- **Stop-the-bleeding:** kill `cli/web`.
- **Assess:** check `users.last_login_at` for unexpected logins; check the reverse-proxy
  / router access logs; confirm the bind address (default `127.0.0.1:8000`).
- **Recover:** fix the exposure path (remove the port-forward / fix the reverse-proxy /
  re-bind to loopback); **rotate `WOBBLEBOT_WEB_SESSION_SECRET`** (invalidates all
  sessions). If exposure was intentional, stand up HTTPS first —
  [`docs/deploy/reverse-proxy.md`](../deploy/reverse-proxy.md).
- **Document:** the exposure path + when it opened.

## 4. gitleaks finding in git history

A secret was committed before the pre-commit hook existed (or slipped past it).

- **Detection:** a `gitleaks detect` history sweep (the quarterly phase-end-audit item)
  flags it; or someone reports it.
- **Stop-the-bleeding:** treat the secret as **compromised right now** — git history is
  clonable, and on a public repo it's already out. **Rotate it immediately** (scenario 1/2).
- **Assess:** which commits, how far back, is the repo public, is the value still live.
- **Recover:** **rotation is the real fix** (it makes the leaked value worthless). *Then*
  decide: `git filter-repo` to scrub history (followed by a coordinated force-push that
  breaks every existing clone) vs document-and-move-on. For a solo public repo where the
  value is already rotated, document-and-move-on is often the pragmatic call.
- **Document:** the secret type, commit range, rotation date, and the rewrite decision.

> The `.githooks/pre-commit` hook (gitleaks + PII + author-identity) prevents **new**
> leaks — install it on a fresh clone via `scripts/install-hooks.*`. This scenario is
> only for history that predates the hook.

## 5. Operator-personal info exposed

A personal email or home-directory path is in a tracked file that's public or about to be.

- **Detection:** the pre-commit PII patterns match in a tracked file; or a PII sweep
  (phase-end-audit) flags one; or you notice it in a screenshot/diff.
- **Stop-the-bleeding:** if it's not yet pushed, scrub it and re-stage — the pre-commit
  hook blocks the commit until clean. If it's already public, move to Assess.
- **Assess:** identify the file:line via the PII patterns (`[/\\]Users[/\\]...`,
  personal-email domains). Is it **identity-bound** — an email tied to a credential or a
  GitHub identity?
- **Recover:** scrub the file; if it's in history and sensitive, `git filter-repo` (same
  force-push caveat as scenario 4); rotate any identity-bound credential. Switch the git
  author identity to a noreply alias if a personal email leaked into commit metadata
  (`git config user.email '<id>+<login>@users.noreply.github.com'`).
- **Document:** what leaked + the remediation.

## 6. Bot behavior diverges from expected

A cap isn't enforced, an order lands at a wrong price, the advisor recommends nonsense,
or orders are orphaned on Kraken.

- **Detection:** a daily-check anomaly (an unexpected fill / price / cap breach); a
  `level=error` or `level=critical` notification in Discord or the web audit log; or
  Kraken open orders that don't match the grid layout.
- **Stop-the-bleeding:** freeze the engine — `cli/live` SIGINT. The `finally` block
  cancels every open order on a clean stop; the **ADR-021 dead man's switch** is the
  backstop that auto-cancels if the daemon can't (crash / power / network loss). For a
  daemon that's stuck or already dead, use the soak-runbook's **process inventory** to
  find survivors before relaunching.
- **Assess:** capture the relevant `data/*.db` rows + logs; compare against expected
  behavior (the cap arithmetic, the grid geometry, the advisor's bounds). Is it a code
  defect or a config value?
- **Recover:** follow the soak-runbook's
  [**Abort + restart procedure**](v1.0-soak-runbook.md#abort--restart-procedure) —
  verify Kraken state, manually cancel any strays in the Kraken UI, re-anchor if BTC has
  drifted from the persisted `grid_state`, fix-in-main as a focused commit, restart.
- **Document:** file as a soak-surfaced defect — its own commit + a test-for-the-bug.

## Quick reference

| Incident | Stop first | Then |
|---|---|---|
| Key/secret compromised | the affected daemon | rotate the credential → restart → re-run `cli/preflight` for Kraken keys |
| Unexpected withdrawal | `cli/harvest` | audit `transfer_results` vs Kraken; rotate harvest key + contact Kraken if not yours |
| `cli/web` exposed | `cli/web` | fix the exposure + rotate the web session secret (kills all sessions) |
| gitleaks history finding | nothing running — the leak is past | **rotate the secret**, then decide history-rewrite vs document |
| Personal info exposed | the commit (hook blocks it) | scrub; filter-repo if in history + sensitive; rotate identity-bound creds |
| Bot misbehaving | `cli/live` (SIGINT) | soak-runbook Abort + restart; file as a defect + test |
