# WobbleBot v1.0 — Known Limitations

*This document captures the v1.0 boundary: what we explicitly did
NOT ship and why. It is a snapshot of the project as of the
v1.0.0 tag — items here are not promises about v1.1+, just an
honest description of what v1.0 is and isn't.*

*Companion document: [`v1.0-future-improvements.md`](v1.0-future-improvements.md)
lists v1.1+ candidate work.*

## Architectural boundaries (ratified by ADRs)

### LLM is advisory only (ADR-002)

The Strategy Advisor produces JSON recommendations; it never
executes trades, never initiates transfers, and never hits Kraken
directly. Auto-application is gated by `cli/apply --commit` (an
operator-initiated tool, ADR-012); the advisor cannot bypass this.

**Why:** the v1.0 risk model treats LLMs as inputs to operator
judgment, not as autonomous agents. An LLM hallucination on the
quant role can't cost money in v1.0; it can only produce a
suggestion the operator declines.

**What's deferred:** a `cli/auto-tune` daemon that polls the
advisor and applies suggestions within configured bounds. Stage
3.4b's auto-apply gate already implements the bounded-application
logic; the daemon wrapper around it is the v1.1 candidate.

### Harvester is the sole transfer authority (ADR-003)

Only `cli/harvest --execute` calls Kraken's `/0/private/Withdraw`.
The Bot Core (trading), Strategy Advisor (LLM), and Web UI all
*lack* withdrawal authority by construction. Kraken enforces this
at the API-key permission level: the trade key has Withdraw scope
OFF; the harvester key has Withdraw scope ON.

**Why:** financial-power fragmentation is the project's headline
safety property. No single module mistake can move money without
operator approval going through the harvester gate's seven defense
layers.

**What's deferred:** automated harvester execution (currently
operator must approve each proposal via `--execute <proposal-id>`).
v1.1 candidate behind appropriate guardrails.

### No separate banking adapter (ADR-004)

The harvester uses Kraken's withdrawal API. There is no
`BankingPort`, no Plaid integration, no separate ACH layer. ACH
deposits from bank to Kraken are operator-initiated from the bank
side; the harvester refuses `bank_to_exchange` proposals at
defense layer 3 with an operator-facing message pointing them to
Kraken Pro's manual deposit flow.

**Why:** Kraken's withdrawal API is the only path that needs
software; deposits already work in the operator's bank app.
Adding a banking adapter is complexity without product value.

**What's deferred:** nothing currently planned. If Kraken ever
exposes an API-initiated deposit path, v1.1 may add it.

### Engine reconciliation at startup only (ADR-018)

`apply_reconciliation` runs once per `cli/live` / `cli/shadow`
startup, between storage-open and signal-handler install. It does
NOT run mid-session, NOT on a schedule, NOT on demand. Exchange is
authoritative: storage-only orders get marked canceled; exchange-
only orders get logged but never adopted into storage.

**Why:** mid-session reconciliation would race the engine's own
order-placement path; per-tick reconciliation would add latency to
every tick for the rare drift case. Startup reconciliation handles
crash-recovery (the primary failure mode) without those tradeoffs.

**What's deferred:** harvester reconciliation. The same pattern
could apply to the `transfer_proposals` + `transfer_results` tables
to detect pending withdrawals Kraken completed since last harvest
session. Phase 8.1 design doc decision 7 explicitly deferred this
to v1.1.

**Also deferred — fill-vs-cancel disambiguation.** The reconciler
matches storage-open against exchange-open by `exchange_id` and
classifies the diff into "storage-only" (mark canceled) and
"exchange-only" (log, don't adopt). It does **NOT** query Kraken's
closed-orders or trade-history endpoints to determine whether a
storage-only order actually FILLED (vs. was canceled). The v1.0
behavior is: storage-only → marked canceled regardless of cause.

**Operational consequence:** if cli/live crashes (or the host
loses power) WHILE an order is filling, on next restart the
reconciler will mark the filled order as canceled and **no
counter-order will be placed** — the BTC inventory ends up
orphaned from the grid strategy until the operator manually
intervenes. Surfaced during the v1.0 soak's first overnight
outage (thunderstorm-induced power loss 2026-05-19; one $10 buy
filled while cli/live was down, no counter sell placed on
restart).

**Mitigation operator can apply post-outage:** either (a) clean
restart from current state (cancel remaining open orders, delete
grid_state row, restart — re-anchors and includes the orphaned
BTC in available SELL collateral), or (b) manual catch-up
(place the missed counter sell directly via Kraken Pro, then
restart — engine will log the orphan but it does the job).

**v1.1 candidate:** reconciler queries Kraken's
`/0/private/ClosedOrders` (+ optionally `/0/private/TradesHistory`)
for storage-only orders, distinguishes fill from cancel, and
replays the counter-placement logic for missed fills. Same
pattern as the engine's `_detect_fills` path, just running once
at startup instead of per-tick.

### Web UI is server-rendered (ADR-016)

FastAPI + Jinja2 + HTMX. No React, no Vue, no Node build pipeline,
no SPA. Mutations cross `pending_commands` rows like Discord
mutations do (ADR-013); the web UI cannot bypass the
confirm-before-execute gate.

**Why:** a server-rendered stack matches the project's "single
deployable artifact" goal and avoids the operational tax of a
JavaScript build chain on a Synology NAS. HTMX gives interactive
flair where it matters (status card refresh) without bringing the
SPA's complexity.

**What's deferred:** richer charts (price history line plots,
PnL curves). Server-rendered SVG via a small charting library is
the v1.1-compatible path; an SPA is out of scope.

### Single-operator web auth (ADR-017)

The web UI authenticates with a bcrypt-hashed password + signed
session cookie. One user row in production. No SSO, no multi-user
roles, no audit trail of which user did what (the operator IS the
audit trail).

**Why:** v1.0 is a single-operator project. Roles / SSO / multi-
user audit would be complexity without value. The web layer is
LAN-only by default (binds 127.0.0.1; operator's reverse proxy
fronts it); attack surface is limited to operator-controlled
hosts.

**What's deferred:** team operation, SSO, role-based access. v1.1+
if the project ever grows past single-operator scope.

## Operational boundaries

### No remote backup destinations in v1.0

`services/backuper.backup_database_locally()` writes to a local
path (`data/backups/<stem>-YYYYMMDD-HHMM.db`). The
`BackupDestination` Protocol is declared so v1.1 can add S3 /
rclone / SFTP variants, but no implementation ships in v1.0.

**Why:** local backups cover the primary risk (corrupted DB
mid-session). Off-host backup is operator infrastructure (their
NAS already replicates volumes, presumably) — codifying it in
v1.0 would force an opinion the operator may not share.

**What's deferred:** S3 backup destination (named "earliest v1.1
candidate" in `docs/release/v1.0-future-improvements.md`).

### Single-host deployment assumed

The project runs as a set of CLI daemons + a FastAPI web app on
one host. There is no cross-host coordination, no leader election,
no shared-storage discipline beyond "SQLite file on local disk."

**Why:** a single Synology NAS is the operator's target deployment;
distribution is complexity without value at this scale.

**What's deferred:** none planned. If the project ever needs to
scale horizontally, the hexagonal architecture allows new ports
(distributed lock manager, etc.) without rewriting domain.

### CryptoCompare 90-day evaluation still pending

ADR-010 swapped paid news sources (CryptoPanic + Whale-alert) for
free ones (RSS + CryptoCompare) at Stage 3.2.5. The decision
includes a 90-day evaluation window to verify CryptoCompare's free
tier reliability meets the news-role's needs. **Due 2026-08-13.**
Deferred to v1.1 evaluation.

**Why:** the 90-day window hasn't elapsed at v1.0 tag time. The
evaluation is a separate decision, not a v1.0 ship blocker.

**What's deferred:** the evaluation itself + any consequent
swap-out. If CryptoCompare proves unreliable, v1.1 may pull in a
different free source.

## Observability boundaries

### No CI / no automated performance regression detection

The project has no GitHub Actions / no CI runner. Pre-commit hooks
(gitleaks + PII pattern + author identity) run locally; `make
check` is the operator's manual gate. Stage 8.3 decision 8
explicitly rejected a CI performance regression check because CI
runner variance makes it untrustworthy at this scale.

**Why:** the operator runs `make check` manually before commits;
the test suite is fast (~20s); the marginal value of CI is small
relative to the operational tax of maintaining workflows on a
single-operator project.

**What's deferred:** GitHub Actions setup. v1.1 candidate IF the
project gains contributors.

### Profile harness is operator-runnable, not continuous

`tools/profile_storage.py` reports p50/p99 ms per storage
operation, but only when the operator runs it. The soak doesn't
automate latency regression detection.

**Why:** continuous profiling adds complexity (where do the
numbers go? when do they alert?) for a single-operator deployment
where the operator can spot-check by running the harness.

**What's deferred:** automated profile-on-startup + alert on
regression. v1.1 if the soak surfaces this need.

### No structured metrics export (Prometheus / Statsd)

The project logs structured records via Python's logging module
(`extra=` dict on each call); `--log-format json` makes them
machine-readable. There is no Prometheus exporter, no metrics
endpoint, no time-series scrape target.

**Why:** the web UI's cost dashboard reads from the `llm_calls`
SQLite table directly; the status dashboard reads `orders` /
`trades` directly. SQLite is the time-series store for v1.0; a
metrics layer is duplication.

**What's deferred:** a `/metrics` endpoint that exports counters
(orders placed, fills, fees paid, harvester proposals) in
Prometheus format. v1.1 candidate IF the operator wires Grafana.

## Tooling boundaries

### Diagnostic tools live under `tools/`, not `cli/`

`tools/show_pending.py`, `tools/show_proposals.py`,
`tools/show_transfers.py`, `tools/show_metrics.py`,
`tools/show_llm_costs.py`, `tools/show_suggestions.py`,
`tools/run_advisor.py`, `tools/run_moe_check.py`,
`tools/run_cloud_check.py`, `tools/first_real_trade.py`,
`tools/profile_storage.py` — eleven operator-runnable diagnostics.

These are NOT counted as "operator entry points" in the running
total (which is 15 cli/ daemons). The convention is established:
`cli/` is for operator-facing operational entry points (daemons,
long-running workflows); `tools/` is for one-shot inspection or
diagnostic scripts.

**Why deferred from cli/:** they're not daemons. Promoting them
would conflate "operator entry point" semantically.

### No `cli/audit` or `cli/forensics` daemon

The audit trail lives in SQLite (`pending_commands`, `llm_calls`,
`notifications`, `conversation_turns`, `transfer_proposals`,
`transfer_results`, the web UI's audit-log view). There is no
single CLI that tails all of them or runs a structured audit
report.

**Why:** the web UI's audit-log view (Stage 7.4) handles human
review; the `tools/show_*` scripts handle ad-hoc CLI inspection.
A unified `cli/audit` would duplicate both.

**What's deferred:** if the operator wants a daily "what
happened" summary emailed or Discord-DM'd, v1.1 candidate.

### No alerting layer beyond Discord notifications

`cli/live` + `cli/harvest` write `notifications` rows;
`cli/operator`'s forwarder posts them to Discord. There is no
PagerDuty, no email, no SMS, no fallback channel if Discord is
down.

**Why:** Discord is the operator's primary attention surface; the
hierarchy is enough for v1.0. A multi-channel alerting layer adds
complexity for a single-operator project.

**What's deferred:** PagerDuty / email fallback. v1.1 candidate
IF Discord-down has been a real problem during the soak.

## Process boundaries

### No release-branch model

The project uses trunk-based development on `main`. No
`release/1.0` branch, no `develop` branch, no GitFlow. Tags mark
releases; commits between tags are the changelog.

**Why:** single-operator project, no parallel feature lines, no
backports. Branch overhead would be cargo cult.

**What's deferred:** none. The model works.

### No semver pre-release tags

v1.0.0 is tagged once; there's no `v1.0.0-rc1` / `-rc2` / etc.

**Why:** the soak IS the release candidate phase. If the soak
finds a defect, we fix it in main and tag v1.0.0 once. If it finds
nothing, we tag v1.0.0 cleanly.

**What's deferred:** for v2.0 or other major bumps, pre-release
tags may make sense. For v1.0, they don't.

### No backwards-compatibility shims in v1.0

v1.0 is the first tagged release. There is no v0.x to be backwards-
compatible with. Schema migrations from v1.0+ onward will use the
existing ALTER TABLE pattern (Stage 3.4a's `expert_opinions`
column add is the precedent).

**Why:** v1.0 is the schema starting point, not a transition state.

**What's deferred:** none. v1.1+ migrations will follow the
established pattern.

## Schema + migration notes

v1.0 ships with these SQLite tables (operator-side files):
`orders`, `trades`, `balance_snapshots`, `balance_entries`,
`grid_state`, `price_snapshots`, `news_items`,
`advisor_suggestions`, `applied_suggestions`, `transfer_proposals`,
`transfer_results`, `pending_commands`, `notifications`,
`conversation_turns`, `llm_calls`, `users`. Sixteen tables.

Future schema changes will be ADDITIVE (new columns with DEFAULT
or NULL, new tables) wherever possible. Destructive changes
(column drops, rename) require a major-version bump or a
documented migration path with rollback.

## Soak-window boundary

This v1.0.0 was tagged after a multi-week operator-driven soak per
the [`v1.0-soak-runbook.md`](v1.0-soak-runbook.md). The runbook
describes what the soak verified (engine reconciliation green on
every startup, no orphaned orders on Kraken, harvester proposals
match expected band semantics, notifications flow end-to-end, etc.).

The soak does NOT verify behavior under conditions the runbook
didn't include — see the runbook's "out of scope" section for the
explicit list (multi-coin grid behavior under high volatility, web
UI under more than one concurrent operator session, harvester
under proposal rejection-then-resubmit, etc.). Those are v1.1+
test surfaces.
