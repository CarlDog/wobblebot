# WobbleBot v1.0 — Soak Runbook

*Operator-facing. Describes what to run, what to watch for, and
what counts as "soak passed" before tagging v1.0.0. The soak is
the release-candidate phase: there's no `v1.0.0-rc1` tag — the
soak IS the rc phase.*

*Companion documents:
[`v1.0-known-limitations.md`](v1.0-known-limitations.md) describes
the v1.0 boundary; [`v1.0-future-improvements.md`](v1.0-future-improvements.md)
lists v1.1+ candidates.*

---

## Why a soak

Stage 8.3 closed all behavior + performance work for v1.0. The
codebase has been behavior-stable since Stage 8.1 closed (the
shadow-session shutdown bug from 2026-05-18 was the last surfaced
defect). The soak is the multi-week observation window that
catches things the unit + integration tests don't:

- Engine reconciliation across many real cli/live restarts.
- Long-running daemon stability (memory growth, file handle
  leaks, SQLite WAL checkpoint behavior).
- Harvester proposal generation across multiple harvest cycles.
- Cloud LLM cost-gate behavior under real (not stubbed) traffic
  if cloud providers are configured.
- Discord operator interaction over enough conversation turns
  that conversation_turns table volume becomes interesting.
- Web UI under regular operator use.
- Maintenance worker's VACUUM + prune + backup cycles completing
  successfully on schedule.

The soak does NOT verify everything; see the "Out of scope"
section below.

## Pre-soak checklist

**Run through each item before starting cli/live. Each is a hard
gate — don't proceed past one that fails.**

### 1. Phase 4 Harvester-key separation (ADR-003 load-bearing invariant)

This is the single most important pre-soak check.

- [ ] Log into Kraken Pro → Settings → API keys.
- [ ] Confirm there are at least TWO keys: a **trade key** and a
      separate **harvester key**.
- [ ] On the **trade key**: confirm **Withdraw funds** scope is
      **OFF**. (Required scopes: Query Funds, Query open & closed
      orders & trades, Create & modify orders, Cancel & close
      orders. **Nothing else.**)
- [ ] On the **harvester key**: confirm **Withdraw funds** scope is
      **ON**. (Required scopes: Query Funds, Withdraw funds.
      Optionally Query ledger entries.)
- [ ] Recommended: enable **IP address restriction** on both keys to
      the operator's Synology / home IP.
- [ ] Confirm `.env` has BOTH key pairs set independently:
      ```
      KRAKEN_READER_API_KEY=...           # read-only key (cli/status)
      KRAKEN_READER_API_SECRET=...
      KRAKEN_TRADER_API_KEY=...     # trade key (cli/preflight, cli/live)
      KRAKEN_TRADER_API_SECRET=...
      KRAKEN_HARVESTER_API_KEY=... # harvester key (cli/harvest --execute)
      KRAKEN_HARVESTER_API_SECRET=...
      ```
- [ ] Confirm `cli/harvest` reads the harvester env vars (not the
      trade key): `grep -nE "KRAKEN_HARVESTER" src/wobblebot/cli/harvest.py`
      should return at least one hit referencing `key_var=` and
      `secret_var=`.

**If any of the above fails, do not start the soak.** Mint the
missing key first or fix the .env / code.

### 2. Config recalibrated for soak amounts

- [ ] Run `python -m wobblebot.cli.status` — confirm the operator's
      current USD balance reads correctly.
- [ ] Decide soak balance allocation (recommended: $10–$20 to start;
      operator can increase post-soak).
- [ ] Run `python -m wobblebot.cli.recalibrate --target-balance N`
      (dry-run by default) — review the per-knob delta table.
- [ ] If the proposed scaling looks right, run with `--commit`. The
      tool round-trips `settings.yml` atomically (preserves comments
      + quoting).
- [ ] Open `config/settings.yml` and spot-check the recalibrated
      knobs:
   - `grid.default.order_size_usd` matches expectation
   - `safety.*_usd` caps look conservative
   - `live.max_session_loss_usd` is small enough that hitting it
     would be a meaningful stop
   - `harvester.{min,topup,surplus}_usd` ordering preserved
     (`min < topup < surplus`)

### 3. Schedules sanity-checked

- [ ] Open `config/settings.yml` → `schedules:` block.
- [ ] Recommended soak cadences:
   - `harvest`: `12h` (twice daily proposal generation)
   - `maintenance_vacuum`: `7d` (default)
   - `maintenance_prune`: `1d` (default)
   - `maintenance_backup`: `1d` (default)
   - `operator_ttl`: `5m` (default; TTL expirer for abandoned
     confirmations)

### 4. Preflight clean

- [ ] Run `python -m wobblebot.cli.preflight`. Expected exit code: 0.
- [ ] If preflight fails: read the error. `validate=true` means
      Kraken signed-and-validated every layout order without
      placing — auth, pair, precision, balance, ordermin, costmin
      all checked. A failure here is a real config problem to fix
      before going further.

### 5. Discord (operator interaction) ready

- [ ] `cli/operator`'s required env var present: `DISCORD_BOT_TOKEN`
      (or whatever `OperatorConfig.auth.bot_token_env_var` names).
- [ ] Bot invited to the operator's Discord server with appropriate
      permissions (read messages, send messages, add reactions in
      the configured channel).
- [ ] `OperatorConfig.outbound_channel_id` set to the channel the
      operator wants notifications posted in.
- [ ] User and channel allowlists set in `OperatorAuthConfig`.

### 6. Web UI ready

- [ ] `WOBBLEBOT_WEB_SESSION_SECRET` env var set to 32+ random
      bytes (mint with `python -c "import secrets; print(secrets.token_urlsafe(32))"`).
- [ ] At least one user seeded via
      `python -m wobblebot.cli.web create-user`.
- [ ] Reverse proxy / firewall configured so `127.0.0.1:8000` is
      reachable from the operator's browser (LAN-only by default
      per ADR-016).
- [ ] **If you're exposing cli/web beyond loopback** (LAN access,
      Tailscale, port-forwarded public domain): read
      [`docs/deploy/reverse-proxy.md`](../deploy/reverse-proxy.md)
      and stand up HTTPS termination before the soak begins. The
      default `127.0.0.1` bind never carries cleartext over a
      network; any other bind value does. cli/web emits a startup
      WARNING when bound to non-loopback to flag the exposure.
- [ ] **Encryption-at-rest on the disk hosting `data/`** — enable
      BitLocker (Windows), FileVault (macOS), LUKS (Linux), or
      Synology volume encryption. The SQLite databases under
      `data/` hold financial data (orders, trades, withdrawal
      destinations, conversation turns) in plaintext; OS-level
      disk encryption is the Tier 0 defense against disk theft and
      cold-backup leakage. Free, zero WobbleBot change. See
      [`docs/deploy/encryption-at-rest.md`](../deploy/encryption-at-rest.md)
      for the threat model + the SQLCipher v1.1 path if backup
      destinations cross trust boundaries.
- [ ] **File permissions on `.env` + `data/*.db` + backups** —
      mode `0600` (owner read/write only) on Linux/macOS, `icacls
      /inheritance:r /grant:r owner:F` equivalent on Windows. See
      the "File permissions (Tier 0 companion)" section in
      [`docs/deploy/encryption-at-rest.md`](../deploy/encryption-at-rest.md)
      for the exact commands per OS. Encryption defends content;
      permissions defend access — both belong to Tier 0.

### 7. Backup destination writable

- [ ] `maintenance.backup_dir` (default `data/backups/`) exists and
      is writable.
- [ ] `maintenance.archive_dir` (default `data/archive/`) exists
      and is writable.

### 8. SECURITY.md disclosure path known

- [ ] Operator knows where to file a security report if something
      surfaces during the soak. (`SECURITY.md` is the reference.)

## What to start (recommended config)

The soak is most useful under a low-risk configuration matching
real-world conditions:

- **Single coin**: BTC/USD. Adding more coins multiplies the
  variables; we want to verify the engine, not measure
  cross-coin correlation.
- **Conservative order size**: $1–$2 per order (post-recalibrate).
- **Wide spacing**: `grid.btcusd.spacing_percentage: 1.0` or
  higher. Fills should be infrequent — the goal is to observe
  the bot's behavior over weeks, not to maximize trade count.
- **Harvester enabled** with conservative thresholds. Soak should
  include the harvester loop running.
- **All daemons running** in their own processes / pm2 / systemd
  units / Portainer stacks (operator's choice):
  - `cli/live` (single-symbol multi-coin loop)
  - `cli/observe` (price snapshots)
  - `cli/news` (RSS + CryptoCompare poller)
  - `cli/advise` (advisor daemon)
  - `cli/harvest` (treasury monitor, default daemon mode — no `--execute`)
  - `cli/operator` (Discord bridge)
  - `cli/web serve` (dashboard)
  - `cli/maintenance` (VACUUM + prune + backup worker)

Eight daemons. They communicate via SQLite tables in their
respective databases per the project's ADR-002 firewall design;
no in-process coordination.

## Daily check-in (operator, ~5 minutes)

The soak isn't "set and forget" — the operator's daily glance is
the dominant signal. Five questions to answer each day:

1. **Are all daemons running?** Check process list (or Portainer /
   pm2 / systemd status). If anything died overnight, the journal
   should have a clean exit message; if not, that's a finding.
2. **Did cli/live restart cleanly if it restarted?** Check the
   most recent reconciliation result in cli/live's log. Storage-
   only orders should be marked canceled; exchange-only orders
   should be logged (not adopted). If anything was unexpectedly
   adopted, that's a Stage 8.1 reconciliation regression.
3. **Any unexpected notifications?** Check Discord + web UI's
   audit log. Per-tick fills + cap notifications are expected;
   anything labeled `level=error` or `level=critical` is a
   finding.
4. **What did the harvester propose?** Check
   `python tools/show_proposals.py --since-hours 24`. Proposals
   should match the band semantics defined in `HarvesterConfig`
   (deficit → no proposal, top-up band → bank_to_exchange,
   hold band → no proposal, surplus → exchange_to_bank).
5. **Did maintenance run?** Check log for `cli/maintenance`
   completing its vacuum / prune / backup cycles on schedule.
   `data/backups/` should grow daily (and prune past
   `keep_n_daily`).

## Things to specifically watch for

These are the "if we see this, fix it before tagging" list:

### Hard stop (abort soak, fix in main, restart)

- **Engine reconciliation adopts an exchange-only order into
  storage.** ADR-018 forbids this. Storage-only → cancel, exchange-
  only → log only. If it happens, that's a regression in
  `services/reconciler.py`.
- **Harvester proposal generation skips a cycle silently.** Either
  it proposed something (logged) or it didn't and logged the
  reason (deficit / hold / day-cap exceeded). Silence is a bug.
- **`cli/live` shutdown leaves orders open on Kraken.** The
  cleanup discipline in the `finally` block must cancel every
  open order regardless of shutdown reason.
- **Any `level=critical` notification.** These are reserved for
  the harvester-execute path (money moved); during the soak the
  harvester runs in daemon mode (no `--execute`), so a critical
  shouldn't fire at all.
- **mypy / pylint / test suite regression on main during a hotfix.**
  Pre-commit hooks should catch it; if anything slips, treat as a
  hard stop.

### Soft watch (note for v1.1 backlog, don't necessarily fix)

- **Engine tick latency** exceeding 1s consistently. Stage 8.3.C's
  `tools/profile_storage.py` on the operator's hardware will give
  baseline; if ticks exceed that significantly during real
  operation, queue for v1.1 (caching / async parallelism per
  future-improvements Group 1).
- **SQLite file growth pattern**. Maintenance worker should keep
  things steady; if `data/*.db` grows monotonically without
  plateaus matching VACUUM cadence, queue for investigation.
- **LLM cost gate behavior** if cloud providers are configured.
  Daily / session caps must hold; if they trip frequently, the
  caps may need calibration (operator-side knob, not v1.0 bug).
- **Discord rate-limit messages** in cli/operator's log. The
  notification forwarder shouldn't generate enough traffic to
  trigger Discord's rate limits at default cadences; if it does,
  the forwarder poll interval needs tuning.

### Information only (note, don't react)

- **First live trade fee verification.** Stage 2.3 documented 0.40%
  taker / 0.26% maker. If a soak-window live trade lands and the
  fee differs, that's information for the audit's quarterly
  cadence — not a soak-blocker.

- **`shutdown hung beyond timeout; forcing exit` WARNING.** As of
  v1.0 (`49e53a7` / `8a85cbd` / `516f4f8` and siblings, 2026-05-23)
  every `cli/*` daemon's resource-close phase is wrapped in
  `safe_shutdown(timeout_seconds=10)`. When a `close()` hangs on a
  stuck file descriptor or socket, the helper logs a WARNING
  naming the in-progress phase (e.g.
  `phase=close_kraken_adapter`) and calls `os._exit(1)` to release
  the terminal. This replaces the pre-v1.0 3-minute hang pattern
  that required `Stop-Process -Force`. **Seeing this WARNING is the
  protection firing as designed, not a defect** — capture the
  phase name for the bug report (the stuck cleanup is the actual
  defect; the WARNING is the recovery).

## What counts as "soak passed"

There is no Claude-mandated duration. The operator decides based on
observed behavior. As a guideline:

- **Minimum useful soak**: ~2 weeks of multi-day cli/live sessions
  (with reconciliation exercised across restarts), at least one
  full cli/maintenance cycle (so VACUUM and backup have both run),
  at least one cli/advise cycle producing a non-trivial suggestion,
  at least one harvester daemon-mode proposal (even if it's just
  "deficit").
- **Comfortable soak**: ~4 weeks with the above plus at least one
  cli/advise → cli/apply --commit cycle (operator-initiated, dry-
  run first), plus enough Discord conversation turns that the
  conversation_turns table has interesting volume.

If the operator finds nothing on the "hard stop" list during the
chosen window and the soft-watch items either didn't materialize
or are acceptable for v1.0, the soak has passed and Stage 8.4.F
release ceremony can run.

## Out of scope for the soak

The soak deliberately doesn't verify:

- **Multi-coin grid under high volatility.** Single coin
  (BTC/USD) keeps the variable set bounded.
- **Web UI under more than one concurrent operator session.**
  Single-operator v1 per ADR-017.
- **Harvester under proposal rejection-then-resubmit.** The gate
  has seven defense layers; testing every combination is a v1.1+
  surface.
- **Recovery from corrupted SQLite database.** SQLite's online
  backup API mitigates the primary risk; deeper corruption recovery
  is a v1.1+ concern.
- **Cloud provider failover.** ADR-015 ratified fail-loudly default;
  the soak might not exercise this if Anthropic / OpenAI / Google
  stay healthy.
- **Engine behavior during a Kraken-side outage** longer than the
  reconciliation retry window. The shadow/live engine swallows
  per-symbol errors at the CLI layer (Stage 2.4 design decision)
  but a multi-hour Kraken outage would surface other concerns.
- **Bot accuracy / profitability over time.** WobbleBot is a
  deterministic engine; whether its grid strategy is profitable
  on a given coin is a strategy question, not a v1.0 ship-blocker.

## Abort + restart procedure

If the operator hits a "hard stop" item during the soak:

1. **Stop the daemon that surfaced the issue.** SIGINT (Ctrl+C in
   the foreground, `pm2 stop <name>` / `systemctl stop <name>` /
   Portainer stop in containerized setups). The `cli/live`
   cleanup `finally` block cancels open orders before exit.
2. **Verify Kraken state**: open Kraken Pro → Trade History →
   Open Orders. There should be no open orders on the soak symbol.
   If there are, manually cancel them in the Kraken UI.
3. **Audit running processes — daemons can die silently.**
   Discovered the hard way on 2026-05-20: `cli/live` crashed but
   `cli/harvest`, `cli/advise`, and `cli/maintenance` were ALSO
   missing from the running process list — they'd died at various
   earlier points without the operator noticing (terminals closed
   in the background, or crashed off-screen). Don't trust "are my
   terminals open?" — run the process inventory:
   ```powershell
   Get-CimInstance Win32_Process -Filter "Name='python.exe'" |
     Where-Object { $_.CommandLine -like '*wobblebot*' } |
     Select-Object ProcessId, CommandLine | Format-Table -Wrap
   ```
   This lists every running wobblebot daemon by PID + command
   line. Any daemon you expect to see and don't = died silently.
   Stop survivors before relaunching:
   ```powershell
   Stop-Process -Id <pid1>, <pid2>, ... -Force
   ```
   then wait ~2s, re-run the audit, confirm empty list before
   starting fresh.
4. **Decide on the anchor.** If `cli/live` died and BTC has drifted
   significantly from the persisted `grid_state.reference_price`,
   manually re-anchor at current price by deleting the row before
   restart:
   ```bash
   sqlite3 data/wobblebot-live.db \
     "DELETE FROM grid_state WHERE symbol_base='BTC' AND symbol_quote='USD'"
   ```
   Engine's `_initialize` path will create a fresh anchor at the
   live price on next startup. (v1.1 candidate: auto-reanchor on
   startup when stale; until then, this is manual.)
5. **Capture the relevant log + DB state** for the bug report.
   The audit-log view in cli/web has the cross-DB story; the
   per-CLI `data/*.db` rows have the raw evidence.
6. **Fix in main**. Per stage-8.4-design.md decision 3,
   soak-surfaced defects ship as focused commits; treat each as
   its own micro-stage with proper test coverage.
7. **Restart the soak**. The soak duration counter resets — the
   minimum useful duration is from the last hard-stop fix, not
   from the original start. This is by design: a soak that
   surfaces a fix mid-way isn't a passed soak.

## When the soak passes

Once the operator decides the soak has run long enough without a
hard-stop finding:

- Notify the operator-driven outcome to the next Claude session
  (e.g. "soak passed YYYY-MM-DD over N weeks, no hard-stop
  findings, ready for 8.4.F").
- Stage 8.4.F (release ceremony) handles:
   - `docs/planning/phase-8-summary.md` writing
   - `pyproject.toml` version bump 0.1.0 → 1.0.0
   - CHANGELOG flip [Unreleased] → [1.0.0] - YYYY-MM-DD
   - `git tag -a v1.0.0` (signed if GPG configured)
   - Roadmap Phase 8 + Stage 8.4 receipts marked ✅

After the tag, v1.0 is shipped. The known-limitations doc is the
boundary; the future-improvements doc is the v1.1+ candidate list.

## Quick reference

**Pre-soak checks**: harvester-key separation, recalibrate, preflight,
Discord, web UI, backup dir.

**Daily checks**: daemons running, reconciliation results,
notification level, harvester proposals, maintenance cycles.

**Hard stops**: bad reconciliation, silent harvester skip, orders
left on Kraken, level=critical, gate regression.

**Pass criteria**: ~2 weeks min, no hard stops, soft-watch items
acceptable for v1.0.

**Out of scope**: multi-coin under high volatility, multi-operator
web, Kraken-outage recovery, profitability evaluation.
