# WobbleBot v1.0 — Future Improvements (catalog index)

*A candidate list for v1.1+. Items here are not promises — each is
a "what we'd do IF the project keeps growing." Each entry names
*what*, *why deferred from v1.0*, and *what would trigger picking
it up*.*

*Companion document:
[`v1.0-known-limitations.md`](v1.0-known-limitations.md) is the
v1.0 boundary; this index points to detailed entries organized by
feature area.*

## Structure

Detailed entries live in [`docs/release/v1.1/`](v1.1/) grouped by
feature area, not by motivation tier. Each entry preserves its
trigger reasoning (soak data / operator feedback / code review /
external triggers) inside the body. Standing rules (which survive
every version boundary) are separated out so they aren't confused
with candidate features.

## Adding new entries

When adding a v1.1+ candidate:

1. Pick the right feature-area file. If no file fits, propose a new
   file rather than dumping into a tangentially-related one.
2. Search the destination file (and the catalog above) for an
   existing entry covering similar ground — prefer augmenting an
   existing entry to creating a near-duplicate.
3. Use the existing entry shape: `### Heading`, then `**What:**`,
   `**Why:**` / `**Why deferred:**`, `**Implementation:**` (when
   applicable), `**Trigger:**`. The shape isn't rigid — entries
   already vary — but the four W's (What / Why / Implementation /
   When-to-act) keep the catalog scannable.
4. Update this index's catalog section with a one-line entry.
5. If the entry has significant architectural implications, save
   a matching OC project memory tagged with the feature area + the
   entry's keyword so semantic search finds it next session.
